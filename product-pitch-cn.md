# VStandby Studio 产品介绍书

## 一句话定位
VStandby Studio 是一个面向孤独观赛人群的 AI 数字人陪伴产品。它不是赛后分析工具，而是一个能和用户一起看比赛、一起反应、一起吐槽、一起讨论的实时观赛伙伴。

## 目标客户
我们的核心用户是：身边没有朋友一起看比赛，也没有稳定的异地观赛搭子，但又希望在看 NBA、电竞或其他赛事时，有人能陪自己同步反应和交流的人。

典型场景包括：

- 一个人在宿舍、家里或通勤时看比赛，想要有人一起喊、一起聊、一起分析。
- 朋友不在同一时区，无法实时连麦看比赛。
- 用户不想进入嘈杂的直播间弹幕，但想要一个懂球、懂自己偏好的陪伴对象。
- 用户喜欢某些球员或队伍，希望 AI 数字人围绕个人偏好提供陪伴式解说。

## 核心产品亮点：用户感知零延迟
传统 AI 观赛产品的问题是：等画面发生、上传、分析、生成，再说话，用户听到时精彩瞬间已经过去了。

我们的方案不是物理意义上的零延迟，而是用户感知上的零延迟：

1. 用户给定直播链接，或用户搜索比赛后由我们找到可暂停、可回放的直播源。
2. 模型输入端观看同一个直播源的最新 live edge。
3. 用户端观看同一个直播源延迟几秒后的位置。
3. 模型在用户看到慢动作之前，已经开始检测、截帧、分析和准备话术。
4. 当用户端画面播放到该慢动作时，数字人已经准备好要说的话，可以像真人一样即时反应。

这让数字人的反应速度从“AI 处理完才说”变成“用户看到时已经准备好说”。

## 为什么选择截图，不录整段视频
当前输出端 LLM 可以读文字和图片，所以我们的慢动作理解链路不再录整段视频，而是走“字幕/解说文本 + 5fps 截图”。

选择截图的原因：

- 截图可以在慢动作发生时持续准备，不需要等待视频片段录制、编码和上传。
- 3 到 5 秒慢动作只需要 15 到 25 张图，体积更小、延迟更可控。
- 输出端 LLM 已经能读图片，关键帧足够支持它理解“发生了什么”和“该怎么反应”。
- 片段结束时，系统已经拿到图片组和文字上下文，可以立刻交给 LLM 生成数字人话术。
- 对实时陪伴来说，我们要优先保证反应速度，而不是保留完整视频文件。

结论：主链路只做截图，不做整段视频录制。

## 直播源策略：同源双时间点
最有说服力的方案不是用户上传视频，而是直接接入直播。

我们优先寻找支持暂停和回放的直播源，也就是 DVR live stream。这类直播在用户暂停后继续播放时，不会强制跳到最新直播点，而是从刚刚暂停的位置继续。

这正好适合我们的产品逻辑：

- 模型端播放最新 live edge，也就是比赛真正实时发生的位置。
- 用户端播放 live edge - N 秒的位置，也就是同一场直播的延迟版本。
- N 秒由系统平均响应速度决定，例如 3 到 8 秒，确保模型能在用户看到精彩瞬间前完成准备。

直播源可以有两种入口：

- 用户粘贴直播链接。
- 用户搜索比赛，我们网站帮助寻找支持暂停/延迟播放的直播源。

如果直播源支持 DVR，我们直接基于同一直播源做双时间点播放。如果直播源不支持 DVR，再 fallback 到后端 ring buffer，由我们自己缓存最近几秒画面制造延迟。

## 新工作流：两条线
### 1. “伪”零延迟线
这条线负责让数字人在用户看到精彩瞬间前，把该说的话准备好。

1. 用户提供直播链接，或搜索比赛后由平台匹配可暂停/可回放直播源。
2. 模型输入端接入同一直播源的最新 live edge。
3. 用户端看到的是同一直播源延迟后的播放位置，延迟几秒取决于我们的平均响应速度。
3. OpenCV 在模型实时流里持续检测慢动作回放。
4. 检测到慢动作后，系统开始每秒截取 5 张图片。
5. 慢动作片段结束时，把字幕/解说文本和截图组发送给输出端多模态 LLM。
6. LLM 生成数字人候选话术。
7. 决策机制根据用户偏好判断说还是不说。

这条线的关键是：模型比用户提前看到直播，所以数字人在用户视角里像是“即时反应”。

### 2. 实时互动线
这条线负责处理用户主动说话。

1. 主模型持续识别用户语音输入。
2. 用户说话后，系统快速语音转文字。
3. 判断用户说的话是否发生在当前慢动作片段附近。
4. 判断用户问题是否和已经准备好的数字人话术相关。
5. 如果相关，并且话术已准备好，就直接让数字人说出来。
6. 如果不相关，或者不在慢动作片段内，说明“伪”零延迟线没有准备这类内容，就转接给通用 LLM 实时处理。

## 数字人与用户互动逻辑
数字人不是每个时刻都说话，而是根据用户偏好智能控制输出频率。

用户偏好包括：

- 话多还是话少。
- 喜欢哪个球员。
- 支持哪支球队。
- 喜欢分析型、情绪型还是吐槽型陪伴。
- 是否喜欢数字人主动输出。

每当慢动作出现时，系统都会预先准备好一句或多句候选话术，但不一定全部说出来。

如果用户偏好是话多：

- 数字人可以主动点评慢动作。
- 可以补充战术、球员表现或情绪反应。

如果用户偏好是话少：

- 数字人只在关键慢动作、争议判罚、比分变化时输出。
- 普通片段可以保持安静。

## 用户说话时的处理逻辑
当用户打开语音输入并和数字人说话时，系统走快速二分判断：

1. 语音快速转文字。
2. 判断用户说话发生时是否处在当前慢动作片段附近。
3. 判断用户内容是否和预生成数字人话术相关。

如果在慢动作片段内，并且和已准备好的话术相关：

- 直接把准备好的结果发给数字人。
- 这样数字人的回答会非常快，像真人已经看懂了刚才那一球。

如果不在慢动作片段内，或者虽然在慢动作片段内但问题和预生成话术无关：

- 转接给通用大模型处理。
- 生成普通聊天、知识问答或陪伴式回应。

这个逻辑的核心是：伪零延迟线已经准备好的内容走快路径；没准备的内容走通用 LLM 实时路径。

## 技术架构简述
内部工作流可以拆成六层：

1. 输入源层：用户直播链接/比赛搜索、模型 live edge、用户延迟播放、用户麦克风、比赛数据。
2. 感知层：字幕/解说提取、语音转文字、OpenCV 慢动作检测、5fps 截图。
3. 结构化层：把原始输入整理成摘要、标签、置信度、比分、时间、片段状态。
4. 输出大模型层：读取文字和图片，生成数字人要说的话。
5. 决策层：根据用户偏好判断是否主动输出，以及输出什么语气。
6. 数字人层：把文本转成语音、表情、动作和屏幕互动。

## Pitch 版本
今天很多人其实不是缺比赛内容，而是缺一个一起看比赛的人。

VStandby Studio 要做的是 AI 观赛搭子。它会陪用户看比赛，在慢动作、进球、判罚、关键回合出现时，像真人朋友一样给出即时反应。

我们的关键创新是用户感知零延迟。模型和用户看的是同一个直播源，但模型看最新 live edge，用户看延迟几秒后的位置。模型提前检测慢动作、截取图片、生成候选话术，所以当用户真正看到那个精彩回放时，数字人已经准备好反应了。

这让 AI 不再像一个慢半拍的工具，而像一个真的坐在你旁边、刚好也看懂了这一球的伙伴。

## 当前推荐落地优先级
第一阶段：

- 实现直播双路流：模型实时流，用户延迟流。
- 支持用户粘贴直播链接，并检测是否支持 DVR/暂停回放。
- OpenCV 检测慢动作开始和结束。
- 每秒 5 张图采样，不录整段视频。
- 输出端 LLM 读取字幕/解说文本和截图组，生成一句候选话术。
- 数字人按固定偏好输出。

第二阶段：

- 加入用户偏好：话多/话少、喜欢球员、支持球队、人格风格。
- 用户语音输入快速 STT。
- 片段相关/无关二分路由。
- 片段相关直接调用预生成话术。

第三阶段：

- 沉淀历史片段和用户偏好记忆。
- 数字人动作、表情和语音风格个性化。
